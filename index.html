什么是E-R模型？

ER模型是实体关系模型，是数据库设计中用于描述现实世界概念结构的一种数据模型，主要核心组成为实体（长方形）、属性（椭圆形）和关系（菱形）。

什么是数据库恢复功能？

数据库恢复功能，是指数据库管理系统在发生故障时，能够将数据库恢复到一致状态的能力。恢复功能是数据库可靠性和数据完整性的重要保障。

数据库崩溃必须要保证的事情：

  • 保证事务的原子性：即使系统崩溃，也要保证事务要么完全执行，要么完全不执行。
  • 恢复数据库到一致状态： 确保数据符合所有预定义的约束条件（如主键、外键、唯一性等）。
  • 最小化数据丢失： 尽可能减少因故障导致的数据损失。
  • 提高系统的可用性：快速恢复，减少停机时间。

常出现的数据库异常：

  • 事务故障
    • 事务因运行逻辑错误或人为终止导致的失败。
    • 解决： 撤销未完成的事务。
  • 系统崩溃
    • 如操作系统崩溃、电源故障等导致的DBMS异常终止。
    • 解决： 使用 日志 和 检查点 恢复已提交和未提交的事务。
  • 介质故障：
    • 磁盘损坏
    • 解决： 从备份或归档日志中恢复数据。
  • 人为错误：
    • 如误删数据，错误更新
    • 解决： 日志回滚。

什么是数据库安全性？

数据库的安全性，是指保护数据库免受 未授权访问、数据泄露、 篡改或破坏的一系列技术和措施。它确保了数据库的 机密性、完整性和可用性，即 CIA三要素。

什么是数据库并发控制？

数据库的并发控制，同常与数据库的隔离级别相结合。如果多个事务不加控制的并发执行，可能导致 脏读、不可重复读和幻读 的问题。

一下是事务的隔离级别：

  • 读未提交：读取到未提交的事务，可能导致 脏读、不可重复读和幻读问题。
  • 读已提交：读取已提交的事务，在事务执行中发生数据改变，可能导致不可重复读和幻读问题。
  • 可重复读： 在事务执行过程中，其他事务插入，可能导致出现幻读问题。
  • 可串行化：事务的执行为串行化，不存在并发问题。

需求分析要完成哪些工作？

需求分析为软件开发的关键阶段，主要目的在于 理解、定义和记录用户需求，为后续系统的设计、开发和测试提供依据。

主要工作：

  • 需求收集： 与用户对接，获取用户的真实需求。
  • 需求分类和整理： 分为功能性需求，非功能性需求、业务规则和用户的其他需求。
  • 通过需求分析与建模： 用例图，实体关系图、流程图等
  • 需求确认：通过团队讨论确认需求的完整性。
  • 需求文档化： 将需求整理成正式的需求规格说明书。
  • 需求分析变更整理：整理可能发生变更的流程。

什么是函数依赖，属性之间的函数依赖反映了什么问题？

在关系形数据库中，函数依赖通常描述了一个关系中属性之间的逻辑约束关系（在MySQL中通过数据库三范式来约束）

反映的问题：

函数依赖揭示了关系数据库中 数据的语义约束 和 潜在的设计缺陷。

  • 数据冗余
  • 更新异常
  • 数据不一致

数据库的三范式

第一范式：

  • 确保每一列都是原子的（不可再分），即每个字段只存储单一值，不允许出现重复组或多值现象。
  • 所有行必须具有唯一标识（如主键），不允许完全相同的重复行。

第二范式：

  • 在第一范式的基础上，消除非主属性对主键的 “部分函数依赖”。
    • 完全主键依赖：非主键必须依赖于整个主键，而非主键的某一部分。
    • 部分函数依赖： 非主键仅依赖于主键的一部分（需拆分表以消除）。
  • 适用于复合主键的表

第三范式：

  • 在第二范式的基础上，消除非主键对主键的”传递性依赖“。
  • 即： 非主属性不能依赖于其他非主属性。

集合

全局介绍

集合整体分为两大接口，包括Map 接口和Collection接口，其中Map接口下实现的类包括 HashMap（重点)、LinkedHashMap、TreeMap、Hashtable和ConcurrentHashMap这些接口，而Collection接口下，还有List、Set、Queue和Deque，这些接口主要的类实现包括ArrayList(重点)、LinkedList、Vector、CopyOnWriteArrayList和HashSet以及LinkedHashSet等。

这里面有一些要注意的类，是线程安全的，包括Hashtable、ConcurrentHashMap、Vector、CopyOnWriteArrayList等。

HashMap（重点）

HashMap实现了Map接口，其组成为 数组+链表+红黑树 的形式。

转化条件：

  1. 当数组长度大于64，链表长度大于8，会将链表转化成红黑树（注意：当数组长度小于64时，HashMap会优先通过扩容来减小链表长度）
  2. 当树的大小小于6时，红黑树会退化成链表。

HashMap 初始容量默认为16，2倍扩容，负载因子为0.75。

优化方式： 可通过设置初始值容量来减少频繁扩容带来的性能消耗，设置合适的负载因子。

存储原理：在将数据存储进入HashMap时，程序会根据hashCode（）方法来计算哈希值，通过哈希值来将数据分配到哈希桶中合适的位置，在该位置判断是否存在数据，若存在，则发生哈希冲突，采用拉链法来保存数据，使用时，会通过equal()方法来判断键值是否存在，若相同，则根据哈希值对应的键来修改数据，不进行插入。若不同，则采取拉链法保存数据。

线程问题： 线程不安全

ArrayList（重点）

ArrayList实现与List接口，底层组成采用 动态数组，初始容量为10，无负载因子概念，1.5倍扩容。

扩容机制： ArrayList采用1.5倍扩容，当扩容发生时，ArrayList会创建一个新的数组，新的数组为旧有数组的1.5倍，之后将旧数组的数据复制到新数组中。

优化方式：设置初始值为需要的部分，防止扩容带来的性能消耗。

存储原理：数组保存。

线程问题：线程不安全。

LinkedList

和ArrayList 相比，采用了 链表 的方式来保存数据，便于插入和删除数据。而ArrayList的保存方式，则适合频繁查找的场景。

TreeMap

内部采用红黑树实现，可通过传入的Comparable的定义来进行排序，键是有序的，不允许键为null，但是值可以为null。使用较少，通常使用在需要排序的加密场景。

HashSet

通过HashMap键的唯一性，来完成实现 set 接口不可重复的概念。

HashTable(线程安全)

采用全局锁的方式，通过synchronized 这个锁，来将整个哈希表锁了起来，对所有的操作（增删改查）进行了上锁，通过这种方式保证了线程的安全性。

ConcurrentHashMap(线程安全)

在Java8中出现，采用了 CAS + synchronized 的形式来保证线程的安全性。

优点：

  1. 在操作节点时，若节点为空，则采用CAS的方式来进行无锁写入操作（无锁，效率高）。
  2. 若不为空，则采用 synchronized 来上锁，保证线程的安全性。
  3. ConcurrentHashMap的线程安全方式和HashTable不同，它锁的是一个结点，在进行锁占用操作时，不影响其他节点的操作。（性能更好，并发性能更高）

Vector（线程安全）

基于动态数组实现，与 ArrayList类似，但是它是线程安全的，所有方法都是同步的（与HashTable 实现线程安全的方式，有异曲同工的地方），将所有的方法都加了synchronized 这个锁。

CopyOnWriteArrayList（线程安全）

基于 动态数组 实现，线程安全。它没有加锁，采用的是一个 写时复制 的方式来保证线程的安全性。

写时复制操作，每次在写操作时，复制一个新的数组，在新的数组上进行修改，原数组不变（不影响读操作），当修改完数组后，将修改后的数组做为原有数组。

由于每次写操作都要生成一个新的数组，因此CopyOnWriteArrayList 的写性能是占用资源多的，并且性能低，适合 读多写少 的场景。

Collection.SynchronizedList(工具类)

它是一个工具类，通过加锁的方式，来保证一个线程的安全性。采用的是加全局锁，并发效率低。

Spring

Spring的启动流程

Spring 启动时，是通过 new ApplicationContext() 这个方式来实现的。

首先说一下Spring启动的核心组件。

  1. ApplicationContext: 体现了 Spring 容器，管理 Bean对象，通过依赖注入的方式来管理 Bean 对象之间的关系，可以通过getBean的方法来获取bean对象。
  2. BeanDefinition：封装了所有的bean信息，系统启动时，会优先读取配置信息里的信息，将其保存成BeanDefinition的形式，想要注册Bean需要先注册BeanDefinition。
  3. BeanFactory ：通过读取 BeanDefinition 的信息，通过反射的机制创建Bean实例。

流程：

在启动Spring时，通过读取配置信息，将Bean 信息保存成BeanDefinition的形式，将多个BeanDefinition保存在map中。

之后通过BeanFactory来创建Bean对象。

SpringAOP

切面编程，是一种编程范式，多用于跨领域的关注点（日志记录、安全检查、事务管理）与业务逻辑分开，通过切面。

动态代理：

主要包括两种，JDK动态代理和CGLIG动态代理。

JDK动态代理：Spring默认的动态代理，通过接口和反射机制实现，要求，类必须有实现的接口。

CGLIB动态代理： SpringBoot2默认的动态代理，通过子类继承的方式来实现动态代理，要求，类必须可以有子类。

AOP相关术语：

    5. 环绕通知
  3. 切点
  4. 目标对象
  5. 代理
  6. 织入

切面的实现，需要通过@Aspect 来注释类，并在类内对方法上添加注解来选择通知的方式和连接点的对象。

Spring事务

事务的隔离级别：

  • 默认：默认隔离级别是 读已提交
  • 读未提交
  • 读已提交
  • 可重复读
  • 可串行化

事务的传播行为，定义管理事务的边界

  • required（默认）： 需要，有则直接使用事务，没有则创建新的事物。
  • required_new: 需要新的， 始终会创建新的事物来执行。
  • supports：支持，如果有事务，使用事务运行，如果没有则不使用事务运行。
  • not_supports: 不支持，始终以非事务的方式来执行任务。
  • mandatory： 必须， 必须要有事务，不然会报错。
  • never： 从不， 必须以非事务的方式运行，不然会报错。
  • nested： 嵌套，如果当前事务存在，则在嵌套事务中执行。

事务的失效：

首先，事务的使用需要用到一个注解 @Transactional.

  1. rollbackFor没有设置对，在使用 @Transactional时，默认配置为只能捕获RuntimeException 和 Error异常，其他异常无法处理。
  2. 异常被捕获，在方法内部，使用 try-catch对异常的类进行了捕获，日志无法获取异常，这样就不会回滚。
  3. 在类内部进行调用，没有使用到动态代理，不开启事务。
  4. Spring事务管理器判断该方法不是public方法时，不支持事务。
  5. 该方法通过final和static修饰，使用CGLIB动态代理，需要子类，final无法子类化，static方法属于类

SpringMVC启动流程

核心组件

  • Dispatcherservlet: 它是SpringMVC 的核心组件，他是一个Servlet，负责接收HTTP请求并调度请求到合适的处理器。
  • @Controller 和 @RequestMapping： 前者用于标记控制类，后者则用于定义控制类中的方法和URL请求的映射关系。
  • ModeAndView：控制返回的数据和视图封装在ModelAndView对象中。
  • ViewResolver（视图解析器）：它负责将逻辑视图名称解析为物理视图，Spring提供了多种视图解析器。
  • @RequestParam和@PathVariable： 用于获取请求参数，后者用于获取URL中的路径变量

请求流程

  1. 当http请求发送到后端时，会先进入Dispatcherservlet 中。
  2. 它会根据请求路径从HandlerMapping 中获取对应的控制器。
  3. 调用控制器Controller
  4. 控制器会直接走到后端，从后端逻辑中获取信息。
  5. 将获取的信息返回（ModelAndView）到Dispatcherservlet。
  6. 它会通过ViewResolver来处理视图映射。
  7. 把信息传给视图层，处理view。
  8. 返回结果给客户端。

JWT 是什么？

JWT（JSON Web Token）是一种开放标准（RFC 7519），用于在网络应用间安全地传输信息。它由三部分组成：Header、Payload 和 Signature，通常以紧凑的 URL 安全字符串形式表示。

JWT 的结构

  1. Header：

    • 包含令牌类型（如 JWT）和签名算法（如 HMAC SHA256 或 RSA）。

JWT 的工作流程

  1. 用户登录：用户提供凭证（如用户名和密码），服务器验证凭证并生成 JWT。
  2. 返回 JWT：服务器将 JWT 返回给客户端。
  3. 客户端存储 JWT：客户端通常将 JWT 存储在本地存储或 cookie 中。
  4. 发送 JWT：客户端在每次请求时发送 JWT，通常在 Authorization 头中。
  5. 服务器验证 JWT：服务器验证 JWT 的签名和声明，如果有效则处理请求。

WT（JSON Web Token）的刷新机制用于在访问令牌过期后获取新的令牌，以延长用户的认证状态，而无需重新登录。

核心概念

  1. 访问令牌（Access Token）：
    • 用于访问受保护资源，有效期较短，通常几分钟到几小时。
  2. 刷新令牌（Refresh Token）：
    • 用于获取新的访问令牌，有效期较长，通常几天到几周。

刷新机制流程

  1. 用户登录：
    • 认证成功后，服务器返回访问令牌和刷新令牌。
  2. 访问资源：
    • 客户端使用访问令牌访问资源。
  3. 访问令牌过期：
    • 当访问令牌失效时，客户端使用刷新令牌请求新的访问令牌。
  4. 刷新令牌：
    • 服务器验证刷新令牌，若有效则返回新的访问令牌，有时还会返回新的刷新令牌。
  5. 继续访问：
    • 客户端使用新的访问令牌继续访问资源。

优点

  • 安全性：访问令牌有效期短，减少被盗风险。
  • 用户体验：无需频繁登录，刷新令牌可延长会话。

关于令牌刷新机制

令牌刷新机制需要手动实现，JWT 本身并不提供自动刷新功能。刷新机制的设计是为了在访问令牌（Access Token）过期后，用户无需重新登录，而是通过刷新令牌（Refresh Token）获取新的访问令牌。

1. 为什么需要刷新机制？

  • 访问令牌有效期短：为了安全，访问令牌通常设置较短的有效期（如 15 分钟）。
  • 用户体验：如果每次访问令牌过期都要求用户重新登录，体验会很差。
  • 安全性：刷新令牌的有效期较长，但只能用于获取新的访问令牌，不能直接访问资源。

2. 刷新机制的核心流程

  1. 用户登录：
    • 服务器返回访问令牌和刷新令牌。
  2. 访问令牌过期：
    • 客户端使用刷新令牌请求新的访问令牌。
  3. 服务器验证刷新令牌：
    • 如果刷新令牌有效，返回新的访问令牌（和可选的新的刷新令牌）。
  4. 客户端使用新令牌：
    • 客户端使用新的访问令牌继续访问资源。

3. 刷新机制 vs 长有效期访问令牌

特性 刷新机制 长有效期访问令牌
安全性 高（访问令牌有效期短，泄露风险低） 低（访问令牌有效期长，泄露风险高）
用户体验 好（无需频繁登录） 差（需要频繁登录）
实现复杂度 较高（需要实现刷新逻辑） 低（无需额外逻辑）
令牌撤销 支持（可撤销刷新令牌） 不支持（无法主动撤销）
适用场景 高安全性要求的系统（如金融、医疗） 低安全性要求的系统（如内部测试）

总结

  • 简单任务：使用@Scheduled注解。

关于XXLJOB和@Scheduled+分布式锁

与XXL-JOB的对比

特性 @Scheduled + 分布式锁 XXL-JOB
实现复杂度 较低 较高
分布式支持 需要自己实现锁机制 内置分布式调度支持
任务分片 不支持 支持
故障转移 需要自己实现 内置支持
动态调度 不支持 支持

适用场景 小规模、简单的分布式任务 大规模、复杂的分布式任务

总结

  • 如果你只是需要一个简单的分布式任务调度，并且任务逻辑不复杂，使用@Scheduled + 分布式锁是一个可行的方案。
  • 如果你的项目需要更强大的分布式调度功能（如任务分片、动态调度、故障转移、监控等），建议直接使用XXL-JOB或类似的分布式调度框架。
